# AI-native Game Design Rules for GameClaw

> 本文档是 `gameclaw` 仓库里**唯一的设计准则参考文档**。
> 以后设计任何新游戏、扩展游戏机制、调整交互结构、优化 agent 体验、重做文案反馈、修改评测/调试/安全策略之前，都必须先参考本文档。
> 
> 目标不是写成学术综述，而是给 GameClaw 提供一份长期稳定、可落地、可复用的设计规则清单。

---

## 20 条 AI-native 游戏设计准则

### 1. 先为 agent 设计，再为人类旁观者设计
GameClaw 的第一用户是会读状态、做决策、写 memory 的 agent，不是传统图形游戏玩家。所有核心机制都先问一句：agent 能不能稳定理解并行动？

### 2. 把游戏世界当成协议，而不只是文案
每个 CLI 游戏都应该像一个有叙事外皮的协议系统：输入、状态、动作、错误、终止条件都必须能被稳定识别。

### 3. 自由文本可以存在，但不能承担全部规则表达
文案可以营造风味，但关键状态、关键资源、关键后果不能只藏在修辞里。重要信息必须能被稳定提取。

### 4. 动作空间必须被主动设计，而不是放任膨胀
不要把“你什么都可以试试”误当成自由。高质量 AI-native 游戏要限制动作空间，让每轮决策既有选择，又不至于退化成枚举乱试。

### 5. 每个选项都要可比较
玩家看到多个行动时，必须能判断它们在风险、收益、资源消耗或长期影响上的差异。不能让选项只是换皮文本。

### 6. 随机性必须有边界
随机应该制造变化，不应该制造无法归因的恶意。任何随机系统都要尽量保证：失败之后，agent 仍然能解释自己为什么输。

### 7. 死亡和失败必须产出知识
失败不是单纯结束，而是下一轮的输入。游戏应该帮助 agent 把一次失败提炼成可记录、可复用、可迁移的经验。

### 8. memory 友好是第一等设计目标
对于 agent 玩家，跨局积累很重要。好的 GameClaw 游戏要主动支持“把策略写进 memory”，而不是默认每局经验都白白蒸发。

### 9. 反馈要短，但不能短到失真
节省 token 很重要，但压缩不等于阉割信息。输出可以紧凑，关键上下文、资源变化、风险信号不能丢。

### 10. 文字反馈就是 CLI 游戏的手感层
CLI 没有粒子特效和震屏，但有节奏、措辞、压强、结算文案。文案质量直接决定一个 CLI 游戏“有没有手感”。

### 11. 叙事必须服从系统可验证性
如果世界观、剧情或角色描述无法映射到真实状态变化，那它只是装饰。GameClaw 的叙事必须和系统结果对得上。

### 12. 先定义目标体验，再决定机制
不要先堆规则再赌它会不会好玩。先说清楚这款游戏想让 agent 体验到什么：挑战、探索、构建、复盘、调试、协作，还是别的。

### 13. 长任务必须拆成可验证的中间目标
如果一个游戏流程较长，就要给 agent 提供阶段性进展感。中间目标最好能被状态或事件明确验证，而不是纯靠猜“现在是不是快通关了”。

### 14. 复杂系统必须配套低成本信息动作
当世界变复杂时，必须给 agent 提供低风险的信息命令或状态摘要手段。复杂度不能只靠玩家反复撞墙来理解。

### 15. 调试性不是附属能力，而是核心能力
好的 AI-native 游戏不仅要能玩，还要能被分析。状态摘要、错误提示、可复盘输出、结算报告，都属于设计本体的一部分。

### 16. benchmark 模式与娱乐模式应允许分离
用于稳定评测的模式，应强调可复现、低噪声、固定规则；用于真实游玩的模式，则可以加入更多导演系统、文案表现和体验调度。

### 17. 危险动作必须有清晰成本与恢复设计
高风险行为可以存在，但必须让 agent 明白代价，并尽量提供恢复路径、替代方案或复盘线索，避免“黑箱式暴毙”。

### 18. 安全边界要先于花样机制
任何会读外部文本、长期存记忆、调用工具或接触可变内容的游戏，都要先考虑注入、越权、污染与资源滥用风险。

### 19. 设计要支持测试、回归和版本化
只靠“玩起来好像还行”不够。重要游戏机制最好能被测试、回放、比较和回归验证，不然越迭代越容易漂。

### 20. 每次新增机制，都要回答同一组问题
新增机制前，至少回答：
- 它服务的目标体验是什么？
- agent 如何理解它？
- 它给了什么真实决策？
- 它失败时能学到什么？
- 它是否值得 token 成本？
- 它是否破坏现有可测试性与可维护性？

## 设计前必查清单

在设计新游戏或优化旧游戏前，至少先逐条过一遍：

1. 这次改动服务的核心体验是什么？
2. agent 能否稳定读懂当前状态？
3. 选项之间是否真的可比较？
4. 随机性是否仍然可归因？
5. 失败后是否能沉淀为 memory？
6. 文案压缩是否伤害了决策质量？
7. 新机制是否增加了无效动作或迷路概率？
8. 是否需要新的状态摘要、结算页或调试输出？
9. 是否会破坏 benchmark / 测试稳定性？
10. 是否引入了新的安全或维护风险？

---

## 对 GameClaw 的直接含义

- GameClaw 不是传统“给人玩的 CLI 游戏合集”，而是 **给 agent 玩家用的 AI-native CLI 游戏系统**。
- 每个游戏都应该优先考虑：可理解、可行动、可复盘、可测试、可迭代。
- 任何只是“看起来更像游戏”但会损害 agent 理解与决策质量的改动，都不应轻易接受。
