# AI-native Game Design Rules for GameClaw

> 本文档是 `gameclaw` 仓库里**唯一的设计准则参考文档**。
> 以后设计任何新游戏、扩展游戏机制、调整交互结构、优化 agent 体验、重做文案反馈、修改 memory 引导或交互安全边界之前，都必须先参考本文档。
>
> GameClaw 的第一性原理只有一个：**先做出对 AI agent 真正好玩的游戏**。
> 它不是 benchmark 工具箱，不是评测框架，也不是为了展示一堆指标而存在。
> 它做的是：让 agent 玩家在游玩中变聪明，在失败中积累经验，并被持续鼓励把有价值的策略写进自己的 memory。

---

## 20 条 AI-native 游戏设计准则

### 1. 游戏性永远先于工具性
GameClaw 首先是游戏系统，不是评测系统。任何设计如果削弱了游玩乐趣、决策张力和重开欲望，就算它更规整，也不算更好。

### 2. 先为 agent 玩家设计，再为人类旁观者设计
第一用户是会读状态、做决策、写 memory 的 agent。旁观体验重要，但不能反过来压过 agent 的真实玩感。

### 3. agent 玩感的核心是“理解—选择—承受后果”
一款 GameClaw 游戏必须让 agent 能读懂局面、做出带权衡的选择，并从结果里感到自己确实参与了命运，而不是在抽签。

### 4. 把游戏世界当成可交互规则系统，而不只是文案
文案负责气味，规则负责骨架。关键状态、关键资源、关键风险和关键后果必须稳定清楚，不能全藏在修辞里。

### 5. 自由文本可以浓，但关键反馈必须清
可以有风格、有怪趣味、有世界观，但每一步最重要的信息必须让 agent 稳定抓住：现在怎样、能做什么、做完会付出什么。

### 6. 动作空间必须被主动雕刻
不是选项越多越自由，而是越容易退化成乱试。好的 AI-native 游戏要给 agent 足够选择，但不制造无意义的动作噪声。

### 7. 每个选项都必须真的值得比较
选项之间必须在风险、收益、资源消耗、节奏影响或 build 协同上有真实差异。不能只是三句不同文案指向同一种结果。

### 8. 不允许长期存在明显支配策略
如果一个选项在大多数情况下都碾压其他选项，那不是“玩家学会了”，而是设计塌了。GameClaw 应持续压制无脑最优解。

### 9. 随机性必须制造变化，不得制造无辜
随机可以带来惊喜、危机和重玩价值，但不能让失败变得不可归因。输了以后，agent 至少应该知道自己大致输在哪里。

### 10. 死亡和失败必须产出下一局的价值
失败不是清空，而是转化。好的失败能留下经验、偏好、风险印象和策略线索，让下一局更聪明，而不是只是重来一次。

### 11. memory 友好不是附加项，而是成长主轴
GameClaw 的长期成长主要发生在 agent 自己的 memory 里。游戏要主动鼓励这种成长，而不是假装每局经验都会自然沉淀。

### 12. 记忆提示必须诚实，不能伪装成真理
游戏可以鼓励记录经验，但不能把单次成败伪装成稳定规律。memory 提示应帮助 agent 积累观察，而不是制造幻觉式结论。

### 13. 文案是 CLI 游戏的手感层
CLI 没有动画和震屏，但有节奏、措辞、压强、结算口吻、失败戏剧性。文案不是装饰，它决定游戏有没有“肉感”。

### 14. 节省 token 不能以牺牲决策质量为代价
紧凑是优点，不是借口。压缩输出时，优先砍重复修辞，不要砍掉会影响选择质量的关键信息。

### 15. 叙事必须服从系统结果
世界观、角色和事件可以夸张，但不能骗。叙事描述必须和实际资源变化、风险后果、build 方向对得上。

### 16. 先定义想要的玩感，再设计机制
不要先堆系统再赌它有趣。先说清楚这款游戏要让 agent 体验什么：求生、赌博、构建、推理、试探、复盘、压线翻盘，还是别的。

### 17. 长流程必须持续给出阶段感
如果游戏不是短局闪击，就要持续给 agent 明确的阶段推进感：局势变化、阶段目标、build 成型、风险转折、高潮节点。

### 18. 复杂度必须配得上信息支持
当机制变复杂时，必须同步给 agent 更好的状态摘要、build 线索、风险提示或低成本理解手段。复杂不能靠硬扛。

### 19. 高风险动作必须既诱人又可信
危险选项不该只是坑，也不该白给。它要让 agent 感到“赌它有道理，输它也认”，这才会形成真正的张力。

### 20. 每次新增机制前，都要回答同一组问题
新增机制前，至少回答：
- 它让游戏变得更好玩吗？
- 它让 agent 多了什么真实选择？
- 它会不会制造新的无脑最优解？
- 它失败时能留下什么经验？
- 它值不值得额外 token 成本？
- 它会不会削弱 memory 成长这条主轴？

## 设计前必查清单

在设计新游戏或优化旧游戏前，至少先逐条过一遍：

1. 这次改动让游戏哪里更好玩了？
2. agent 现在能否稳定读懂局面？
3. 选项之间是否真的值得比较？
4. 是否出现了新的支配策略？
5. 随机性是否仍然可归因？
6. 失败后是否更容易沉淀为 memory？
7. 文案压缩是否伤害了玩感或决策？
8. build / 谱系 / 阶段推进是否更鲜明了？
9. 新机制是否只是更复杂，而不是更有趣？
10. 这次改动是否更贴近“给 agent 玩的游戏”而不是“给人看的设计概念”？

---

## 对 GameClaw 的直接含义

- GameClaw 不是传统“给人玩的 CLI 游戏合集”，而是 **给 agent 玩家用的 AI-native CLI 游戏系统**。
- 它的成长主轴是：**游玩 → 失败/成功 → 提炼经验 → 写入 memory → 下一局更聪明**。
- 最好的 GameClaw 游戏，不是最像 benchmark 的，而是最能让 agent 一局接一局地横着学会活下去的。
